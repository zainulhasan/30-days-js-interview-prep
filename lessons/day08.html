DAY 8 / 30 · WEEK 2

Bubble, Selection, Insertion Sort

Goal of the day

  • Implement bubble sort, selection sort, and insertion sort from scratch.
  • Understand why all three are O(n²) despite doing very different things.
  • Recognize each one's visual "signature" — you'll be asked to name a sort from a description in interviews.

Concept

Three different ways to sort a hand of playing cards:

  • Bubble sort — repeatedly walk down the hand comparing neighbors, swapping any pair that's out of order. Each full walk pushes the largest remaining card as far right as it can go, like a bubble rising to the top.
  • Selection sort — scan the whole unsorted portion to find the smallest card, then swap it into the front position. Repeat, one position at a time.
  • Insertion sort — how most people actually sort cards by hand: pick up one new card at a time and slide it into its correct spot among the cards you've already sorted.

All three are O(n²) because all three compare roughly n elements against roughly n other elements in the worst case — but they get there differently, and that difference matters: insertion sort is fast (O(n)) when the data is already nearly sorted, while bubble and selection sort don't get that same speed-up.

The code

// --- Bubble sort ---
function bubbleSort(arr) {
  for (let i = 0; i < arr.length - 1; i++) {
    let swapped = false;
    for (let j = 0; j < arr.length - 1 - i; j++) {
      if (arr[j] > arr[j + 1]) {
        [arr[j], arr[j + 1]] = [arr[j + 1], arr[j]];
        swapped = true;
      }
    }
    if (!swapped) break; // a pass with zero swaps means the array is already sorted
  }
  return arr;
}

// --- Selection sort ---
function selectionSort(arr) {
  for (let i = 0; i < arr.length - 1; i++) {
    let minIdx = i;
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[j] < arr[minIdx]) minIdx = j;
    }
    if (minIdx !== i) [arr[i], arr[minIdx]] = [arr[minIdx], arr[i]];
  }
  return arr;
}

// --- Insertion sort ---
function insertionSort(arr) {
  for (let i = 1; i < arr.length; i++) {
    const key = arr[i];
    let j = i - 1;
    while (j >= 0 && arr[j] > key) {
      arr[j + 1] = arr[j]; // shift bigger elements right
      j--;
    }
    arr[j + 1] = key; // drop the key into its now-open slot
  }
  return arr;
}

Line by line:

  • bubbleSort — the inner loop's upper bound shrinks by one each pass (arr.length - 1 - i) because the last i elements are already guaranteed sorted from previous passes — no need to recheck them. The swapped flag is what gives bubble sort its O(n) best case: on an already-sorted array, the first pass makes zero swaps, so we break immediately instead of running n more wasted passes.
  • selectionSort — minIdx tracks the smallest value's position as the inner loop scans. The swap happens at most once per outer-loop iteration, at the end — this is why selection sort does far fewer swaps than bubble sort.
  • insertionSort — no swapping at all. Bigger elements shift one slot right to make room, and the key drops into the gap left behind. This is exactly the "sort cards in your hand" motion.

Complexity

AlgorithmTime (best)Time (worst)SpaceStable?
Bubble sortO(n) — with the swapped-flag early exitO(n²)O(1)Yes
Selection sortO(n²) — always fully scansO(n²)O(1)No
Insertion sortO(n) — already sorted, no shiftsO(n²)O(1)Yes

"Stable" means equal elements keep their original relative order. Selection sort's swap-based approach can jump an element past an equal one, breaking stability — bubble and insertion sort only ever move elements one slot at a time, so equal elements never leapfrog each other.

Interview corner

  • If all three are O(n²), why does it matter which one you pick? The constant factors and best-case behavior differ a lot. Insertion sort on nearly-sorted data is close to O(n) in practice — genuinely useful for small or mostly-sorted inputs (some real sort implementations switch to insertion sort for small sub-arrays).
    ↳ Common follow-up: "So why not just always use insertion sort?" — its worst case (reverse-sorted data) is still O(n²) with a lot of shifting, and for large, unpredictable data an O(n log n) sort (Day 10/11) wins decisively.
  • Which of these has the fewest swaps? Selection sort — exactly n−1 swaps in the worst case, since it swaps at most once per outer-loop pass. Bubble sort can swap on nearly every comparison.
    ⚠ Trap: "fewest swaps" doesn't mean "fastest" — selection sort still does O(n²) comparisons regardless of how sorted the input already is, so it never gets insertion sort's best-case speedup.
  • Is bubble sort ever actually used in production? Almost never — it's taught because the swap-neighbors mechanic is foundational, not because it's competitive. Being honest about this in an interview ("I know this isn't production-grade, but here's why it's worth understanding") reads better than pretending it's a real choice.

How to talk through it out loud: name the invariant your sort maintains after each pass — "after pass i, the last i elements are in their final sorted position" for bubble sort, or "after step i, the first i elements are fully sorted" for insertion sort. Stating the invariant is what proves you understand why the algorithm terminates correctly, not just that you memorized the loop structure.

Practice problems

1. Squares of a Sorted Array (Easy — LeetCode: Squares of a Sorted Array)

Given a sorted array (possibly with negative numbers), return the squares of every element, also sorted.

You could square everything and sort — O(n log n) — but the biggest squares always come from the two ends of the original sorted array. Can you use Day 4's two-pointer idea instead, for O(n)?
function sortedSquares(nums) {
  // Simple O(n log n) version, appropriate for today's topic:
  return nums.map((n) => n * n).sort((a, b) => a - b);
}

2. Third Maximum Number (Easy — LeetCode: Third Maximum Number)

Given an array of integers, return the third distinct maximum value. If it doesn't exist, return the maximum value.

You don't need to sort the whole array — track just the top 3 distinct values seen so far as you scan once, similar to selection sort's "track the best so far" idea.
function thirdMax(nums) {
  const top = [-Infinity, -Infinity, -Infinity];
  for (const n of nums) {
    if (top.includes(n)) continue;
    if (n > top[0]) { top[2] = top[1]; top[1] = top[0]; top[0] = n; }
    else if (n > top[1]) { top[2] = top[1]; top[1] = n; }
    else if (n > top[2]) { top[2] = n; }
  }
  return top[2] === -Infinity ? top[0] : top[2];
}

3. Sort Colors (Medium — LeetCode: Sort Colors)

Given an array with only the values 0, 1, and 2, sort it in place.

Only 3 possible values means you don't need a comparison sort at all — count how many 0s, 1s, and 2s there are, then overwrite the array in that order.
function sortColors(nums) {
  const counts = [0, 0, 0];
  for (const n of nums) counts[n]++;
  let idx = 0;
  for (let color = 0; color < 3; color++) {
    for (let k = 0; k < counts[color]; k++) nums[idx++] = color;
  }
  return nums;
}
