ARRAYS & HASHING · MEDIUM

Maximum Subarray (LeetCode: Maximum Subarray)

The problem

You're given an array of integers nums — it may contain negative numbers. Find the contiguous subarray (a run of consecutive elements, no skipping around) that has the largest possible sum, and return that sum — not the subarray itself. The subarray must contain at least one number; you can't "opt out" and claim an empty subarray sums to zero.

Example: nums = [-2,1,-3,4,-1,2,1,-5,4] → 6, from the subarray [4,-1,2,1]. A single-element array like [1] just returns 1. And because "at least one number" is a hard requirement, an all-negative array like [-3,-1,-8,-2] returns -1 — the least-negative single element — never 0.

Solution 1: Brute force — every window, with a running sum

The most direct reading of the problem is "try every contiguous window and keep the best sum." The fully naive version of that idea is O(n³): for every (start, end) pair, re-add up all the elements between them from scratch. That's wasteful in a way that has nothing to do with the actual algorithmic idea being demonstrated here — it's just redundant re-summation. So the version below is the improved O(n²) brute force: for each start, walk end forward and maintain a running sum, adding just nums[end] each step instead of re-summing the whole window. Same nested-loop shape, same "try every window" idea, but each window now costs O(1) instead of O(n) — which is what turns this into a clean two-step complexity story (O(n²) → O(n)) against the optimal solution below, instead of a throwaway O(n³) comparison that would only be about fixing an unrelated inefficiency.

The code

function maxSubArrayBrute(nums) {
  let best = -Infinity;
  for (let start = 0; start < nums.length; start++) {
    let sum = 0;
    for (let end = start; end < nums.length; end++) {
      sum += nums[end];
      if (sum > best) best = sum;
    }
  }
  return best;
}

Line by line:

  • best starts at -Infinity, not 0 — the problem requires at least one number in the subarray, so an implicit empty/zero-sum subarray must never be allowed to win.
  • The outer loop fixes start; the inner loop grows end one step at a time.
  • sum += nums[end] is the "running sum" optimization: each new end only costs one addition, instead of re-adding every element from start to end from scratch.
  • if (sum > best) best = sum checks every window's sum exactly once, the moment it's computed.
  • Every (start, end) pair is still visited — that's what keeps this O(n²) despite the running-sum improvement.

Solution 2: Optimal — Kadane's Algorithm, single pass

Every window-sum in the brute force answers the same underlying question over and over: "what's the best subarray sum that ends exactly at this index?" That question has a clean recurrence. Let currentSum be the best sum of a subarray ending at the current index. At each element, there are only two options: extend the previous run by adding this element (currentSum + nums[i]), or abandon that run and start fresh right here (nums[i] alone) — worth doing whenever the previous run had gone so negative that dragging it forward would only hurt. Kadane's algorithm is exactly that one-line decision, made once per index: currentSum = Math.max(nums[i], currentSum + nums[i]). A second running variable, maxSum, remembers the best currentSum seen across the whole scan.

The code

function maxSubArrayOptimal(nums) {
  let currentSum = nums[0];
  let maxSum = nums[0];

  for (let i = 1; i < nums.length; i++) {
    currentSum = Math.max(nums[i], currentSum + nums[i]);
    maxSum = Math.max(maxSum, currentSum);
  }
  return maxSum;
}

Line by line:

  • currentSum and maxSum both start at nums[0], not 0 or -Infinity — the array is guaranteed non-empty, and starting at the first real element correctly seeds both a single-element array and an all-negative one.
  • currentSum = Math.max(nums[i], currentSum + nums[i]) is the entire algorithm: extend the run by adding nums[i], unless the run so far had gone negative enough that starting fresh at nums[i] alone is strictly better.
  • maxSum = Math.max(maxSum, currentSum) runs on every iteration, whether or not this step restarted — the best subarray doesn't have to end at the very last index, so the running best has to be captured as the scan goes.
  • The loop never looks backward past currentSum — that one number already summarizes everything relevant about every earlier index.

This is a genuine asymptotic time improvement, not a constant-factor tweak — Kadane's algorithm drops from quadratic to linear by proving that "the best sum of every window" can be replaced with "the best sum ending at each index," a single running number updated once per element. Space stays O(1) on both sides — the running-sum brute force never allocates a table, it just doesn't visit indices any faster.

Interview corner

Why is it safe for currentSum to summarize the entire scan so far, without keeping any other history?

  • currentSum is defined as "the best sum of a subarray ending exactly at the current index" — that's a complete answer for this index, not a partial one.
  • Any subarray ending at index i is either the single element nums[i], or some earlier subarray extended by nums[i] — and the best "earlier subarray" for that purpose is exactly the best one ending at i-1, which is what the previous currentSum already holds.
  • Nothing further back than i-1 can ever matter once currentSum at i-1 is known — that's what makes O(1) extra state enough.

Why must maxSum be initialized to nums[0], and not 0?

  • The problem requires the subarray to contain at least one number — an empty subarray (implicitly summing to 0) is not a valid answer.
  • Initializing to 0 would silently let that invalid empty subarray win whenever every real subarray sums to something negative — exactly the all-negative-array case, e.g. [-3,-1,-8,-2], whose correct answer is -1, not 0.
  • Seeding both currentSum and maxSum with the first real element instead sidesteps the whole "what does empty even mean here" question.

This solution only returns the maximum sum — how would you adapt it to also return the winning subarray itself?

  • Track one more index alongside currentSum: runStart, the index where the current run began. Reset it to i whenever the algorithm restarts instead of extends.
  • Whenever maxSum updates, also snapshot bestStart = runStart and bestEnd = i — exactly what the visualizations above are doing under the hood to know which window to highlight as "sorted" at the end.
  • This adds O(1) extra bookkeeping — no change to the O(n) time or O(1) space bounds.

Trick question — what does this log?

const nums = [];
console.log(Math.max(...nums));
  • Answer: -Infinity — not undefined, and not a thrown error.
  • Math.max() called with zero arguments (which is exactly what Math.max(...[]) spreads down to) is specified to return -Infinity — the identity value for "the maximum of nothing."
  • Both solutions on this page use Math.max heavily, but never on a possibly-empty spread — that's exactly why the "at least one number" guarantee in the problem statement matters in practice.

⚠ Trap: if you were tempted to "one-line" a solution as something like Math.max(...allWindowSums), an accidentally-empty nums wouldn't throw or warn you — it would silently return -Infinity, a wrong-looking number that's easy to miss in review precisely because nothing crashes.
